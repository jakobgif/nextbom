# Development Reference

Technical reference for contributors to nextbom.

## Tech Stack

- **Frontend**: React 19, TypeScript, Vite, Tailwind CSS v4, shadcn/ui (Radix UI primitives), Zustand, Sonner (toasts)
- **Backend**: Rust via Tauri 2
- **Database**: SQLite via rusqlite (for `.nextbom` database files)
- **Project files**: JSON serialized via serde

## Architecture

### Frontend (`src/`)
- `App.tsx` — root component, listens for `project-changed` Tauri events, renders empty state or project view (accordion-based workflow steps)
- `components/titlebar.tsx` — custom titlebar with Menubar (File/Edit/Help), window controls, unsaved indicator, and all project edit dialogs
- `components/new-project.tsx` — `NewProjectDialog` for creating projects (title, engineer, project_specifics)
- `components/set-string-dialog.tsx` — `SetStringDialog`, generic reusable dialog for editing any single string field; used for title, engineer, project_specifics
- `components/ui/` — shadcn/ui component wrappers (button, input, dialog, accordion, popover, menubar, etc.)
- `src/types/` — TypeScript types generated by `ts-rs` from Rust structs; **do not edit manually**

### Backend (`src-tauri/src/`)
- `lib.rs` — Tauri app setup, `AppState` (holds `Mutex<Option<Project>>`, `Mutex<Option<String>>` for path, `Mutex<bool>` for unsaved flag)
- `commands/mod.rs` — re-exports everything from `project.rs`
- `commands/project.rs` — all Tauri commands: `get_project_state`, `create_project`, `open_project`, `save_project` (supports `save_as`), `close_project`, `set_project_title`, `set_project_engineer`, `set_project_specifics`, `set_database_path`, `import_csv_to_database`
- `data/mod.rs` — re-exports from submodules
- `data/project_data.rs` — `Project` and `ProjectState` structs (serde + ts-rs)
- `data/bom.rs` — BOM SQLite schema, CSV parsing (semicolon-delimited), `BomEntry` struct
- `data/recent_projects.rs` — `RecentProject` and `RecentProjects` structs (implemented but not yet wired to any Tauri commands)

### Communication
- Frontend calls Rust via `invoke("command_name", args)` — Tauri converts camelCase JS args to snake_case Rust params automatically
- Backend emits `project-changed` events (payload: `ProjectState`) after every state mutation
- Both `App.tsx` and `Titlebar` independently call `get_project_state` on mount and subscribe to `project-changed` — no React context; each component manages its own copy

### Capabilities (`src-tauri/capabilities/default.json`)
Explicit permissions: `core:default`, `opener:default`, window controls, `dialog:allow-save`, `process:allow-exit`, `process:allow-restart`. File-open dialogs are covered by `core:default`. When adding new Tauri plugins, add their permissions here.

## TypeScript Types

Types in `src/types/` are auto-generated by `ts-rs` from Rust structs. To regenerate after changing Rust types, run the Rust tests or build. **Never edit these files by hand.**

## Key Data Structures

- `Project` — uuid, title, engineer, project_specifics, database_path, latest_bom_version, last_change (Unix ms timestamp), schema (semver string)
- `ProjectState` — `{ project: Project | null, has_unsaved_changes: boolean }` (event payload and `get_project_state` return type)
- `BomEntry` — `{ part_id: string, designator: string }` (rows in the SQLite BOM table)
- `RecentProject` — file_path, title, last_opened timestamp

## Project File Format

`.nbp` files (JSON, serde pretty-printed). Schema version uses major-version compatibility: a file created with v1.x can only be opened with v1.x software. New project has no file path until first save.

## BOM Working File Format

SQLite `.nextbom` working file with a `bom` table: `designator TEXT PK, part_id TEXT`. CSV import expects semicolon-separated files with a header row; column 0 = part ID (e.g. `CAP00100`), column 1 = designator.

## Parts Database Format

SQLite `.nextdb` file with tables mapping generic part IDs to manufacturer part numbers.

## Running the App

```bash
# Dev mode (hot reload)
npm run tauri dev

# Release build
npm run tauri build

# Frontend only (no Tauri)
npm run dev
```

## Conventions

- Tauri commands return `Result<T, String>` — errors are plain strings displayed via `toast.error()`
- User-cancellation (e.g. "No file selected", "No save location selected") is caught on the frontend and swallowed silently — not shown as an error
- `AppState` mutators: lock the mutex, mutate, `drop()` the guard, then emit the event — never hold a lock while calling `app.emit()`
- Every state mutation emits `project-changed` so the frontend always stays in sync
- `Project::touch()` updates `last_change` to `Utc::now().timestamp_millis()` — call via the setter methods, not directly
- Async Tauri commands use `tauri::async_runtime::spawn_blocking` for blocking file dialogs to avoid blocking the async executor

## Rust Code Quality

### Documentation
All Rust items (functions, structs, enums, methods) must have a `///` doc comment that explains what it does, its parameters, and any non-obvious behaviour. This applies to both public and private items. The goal is that any function can be understood without reading its body.

Avoid comments that will become stale as the code evolves:
- Do not repeat a literal value that already appears in the code (e.g. don't write "default maximum of 10 items" when the body says `Self::new(10)`).
- Do not name specific files or components from other layers (e.g. do not reference `App.tsx` or `Titlebar` in a Rust doc comment) — use role descriptions instead ("frontend components").
- Do not describe *how* something is implemented when the code is self-evident; focus on *what* and *why*.

### Unit Tests
Write `#[cfg(test)]` unit tests for any pure or near-pure Rust logic — functions that transform data, validate input, parse formats, or manage state without requiring Tauri internals (`AppHandle`, `State`, file dialogs). Tests live in a `mod tests` block at the bottom of the same file. Tauri command handlers that depend on `AppHandle`/`State` are excluded; cover those via E2E if needed.

Examples of code that must have tests:
- CSV parsing (`parse_csv`)
- Schema version compatibility checking
- `RecentProjects` add/remove/sort logic
- Any new data transformation or validation function
